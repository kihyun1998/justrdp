# PKINIT Smartcard Provider — Spec Notes

> **Generated by**: `@spec-checker` Step 1 for roadmap §9.6
> **Reference**: RFC 4556 (PKINIT for Kerberos), NIST SP 800-73-4 (PIV)
> **Scope**: minimal spec analysis to fix the `SmartcardProvider` trait
> shape — full PKINIT wire format already implemented and out of scope.

## 1. Where the certificate goes (RFC 4556 §3.2.1)

`PA-PK-AS-REQ.signedAuthPack` is a CMS `ContentInfo { contentType:
id-signedData, content: SignedData }`. The certificate chain lives in
`SignedData.certificates` ([0] IMPLICIT SET OF Certificate).

**Chain rules**:
- **SHOULD**: include enough certs to build a path to a KDC-trusted
  trust anchor.
- **MUST NOT**: include the root CA itself.
- A smartcard typically exposes only its end-entity cert; intermediate
  CAs need to be discovered separately (PKCS#11 `C_FindObjects` or
  pre-loaded by the application).

**Implication**: provider must expose `get_intermediate_chain()` —
returning end-entity alone is insufficient for environments with
intermediate CAs.

**Existing slot**: `cms.rs:80` already handles the certificates
field; we just need to feed it `[end_entity] + intermediates`.

## 2. What gets signed (RFC 4556 §3.2.1)

- `signedAuthPack.signerInfos[0].signature` is the signature.
- Signature input = DER encoding of `AuthPack`, which contains
  `PKAuthenticator` + `clientPublicValue` (DH).
- Algorithm: `sha-256WithRSAEncryption` (OID `1.2.840.113549.1.1.11`)
  in current code (`OID_SHA256_WITH_RSA` hardcoded in `build_signer_info`).

**Current implementation** (`kerberos.rs:357`):
```rust
let signature = rsa_sign_sha256(&config.private_key, &auth_pack_der);
```
`rsa_sign_sha256` internally does `pkcs1v15(sha256(auth_pack_der))`.

**Smartcard split**: SHA-256 is computed on the host; the card
receives the 32-byte digest and applies PKCS#1 v1.5 padding + RSA
decryption with the private exponent. This matches PIV (NIST SP
800-73-4 §3.3.2 GENERAL AUTHENTICATE) and PKCS#11
(`CKM_RSA_PKCS` with pre-hashed input).

## 3. Provider trait shape (final)

```rust
pub trait SmartcardProvider {
    /// End-entity X.509 certificate, DER-encoded.
    fn get_certificate(&self) -> Vec<u8>;

    /// Intermediate CAs in chain order, root excluded.
    /// Empty Vec if the end-entity directly trusts a KDC anchor.
    fn get_intermediate_chain(&self) -> Vec<Vec<u8>>;

    /// PIV SP 800-73-4 §2.4: VERIFY command, once per session.
    /// Subsequent sign_digest calls reuse the cached PIN state.
    fn verify_pin(&mut self, pin: &[u8]) -> Result<(), SmartcardError>;

    /// Sign a 32-byte SHA-256 digest with PKCS#1 v1.5 padding + RSA.
    /// Returns the signature bytes (modulus length).
    fn sign_digest(&self, digest: &[u8]) -> Result<Vec<u8>, SmartcardError>;
}
```

**Out of scope for the trait**:
- Reader enumeration / card selection — these are concrete-provider
  construction-time concerns (Mock = struct literal, PCSC =
  constructor enumerates readers and connects).
- DH keypair generation — purely host-side
  (`PkinitConfig.dh_private_bytes` stays as-is, untouched by the
  smartcard layer).

## 4. Integration points

- **`crates/justrdp-connector/src/credssp/kerberos.rs:357`** — replace
  `rsa_sign_sha256` call with: compute SHA-256 host-side, then call
  `provider.sign_digest(&digest)`.
- **`crates/justrdp-pdu/src/cms.rs:80`** — `build_signed_data` already
  takes `certificates: &[&[u8]]`; pass the full chain.
- **`PkinitConfig`** — add a `provider: Option<Box<dyn SmartcardProvider>>`
  variant or split into two enum variants (raw key vs smartcard).
  Backward compatibility: keep the existing raw-key path working
  unchanged when `provider` is `None`.

## 5. Hash algorithm

Locked to **SHA-256 + PKCS#1 v1.5** for now, matching the existing
`OID_SHA256_WITH_RSA` constant. Future extension (SHA-384, ECDSA) can
parameterize via an enum but is out of scope for §9.6.

## 6. Error handling

`SmartcardError` should distinguish:
- `PinIncorrect` (and remaining tries if known)
- `PinBlocked`
- `CardRemoved`
- `CardNotPresent`
- `CryptoFailure(String)`
- `Other(String)`

PIN handling MUST zeroize the input buffer after use (security-scanner
will check this).
