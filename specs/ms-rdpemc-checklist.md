## 1. Transport (MS-RDPEMC §2.1)

- [ ] Channel type: STATIC virtual channel, registered as "encomsp" (7 ANSI chars + NUL)
      (MS-RDPBCGR §3.1.5.2) — NOT DRDYNVC
- [ ] Channel name constant: ENCOMSP_CHANNEL_NAME = b"encomsp\0" (8 bytes)
- [ ] Multiple MS-RDPEMC messages MAY be concatenated in one SVC PDU;
      receiver MUST loop-parse until all bytes consumed (§3.1.5.1)
- [ ] Prerequisite: "encomsp" SVC transport established before any MS-RDPEMC PDU (§1.5)
- [ ] Client sends a pre-connection PDU per [MS-RDPEPS] §2.2.1 before RDP; MS-RDPEMC has
      no in-band capability negotiation

---

## 2. Capability Negotiation / Versioning (MS-RDPEMC §1.7)

- [ ] MS-RDPEMC has NO in-band capability or version negotiation. No version fields.
- [ ] Implicitly versioned by Windows OS version of sharing manager.
- [ ] Unknown msgType SHOULD be silently ignored (skip by Length) (Appendix A <7>).
- [ ] ALLOW_CONTROL_REQUESTS flag (0x0008) NOT used by Windows (Appendix A <30>).
- [ ] OD_WND_REGION_UPDATE (0x000C) NOT sent by Windows (Appendix A <5>). Must parse if
      received.

## 3. Wire Format — Common Structures

### 3.1 Common Order Header (ORDER_HDR) (§2.2.1)

- [ ] Total 4 bytes.
- [ ] Type: u16 LE, offset 0, range 0x0001–0x000D.
- [ ] Length: u16 LE, offset 2; MUST equal total PDU size INCLUDING the 4-byte header.
- [ ] Inconsistent Length → terminate connection (Appendix A <8>).
- [ ] Trailing extra bytes → tolerate (Appendix A <9–24>).
- [ ] Minimum valid Length = 4 (header-only PDUs).

### 3.2 Unicode String (UNICODE_STRING) (§2.2.2)

- [ ] cchString: u16 LE, offset 0; UTF-16LE character count (NOT bytes). Range 0–1024.
- [ ] String: variable, cchString*2 bytes, UTF-16LE; absent when cchString == 0.
- [ ] Total encoded size: 2 + cchString*2.
- [ ] Receiver MUST validate cchString*2 bytes remain (§3.1.5.2).
- [ ] Terminated at first UTF-16 NUL or at cchString chars.

## 6. Individual PDU Wire Formats

### 6.1 OD_FILTER_STATE_UPDATED (§2.2.3.1) — SM→P, 5 bytes

- [ ] HDR.Type=0x0001, HDR.Length=0x0005
- [ ] Flags: u8, offset 4; FILTER_ENABLED = 0x01
- [ ] On state change receiver SHOULD flush stored app/window data (Appendix A <13>)

### 6.2 OD_APP_CREATED (§2.2.3.2) — SM→P, 12 + cchString*2 bytes

- [ ] HDR.Type=0x0003, HDR.Length=0x000C + cchString*2
- [ ] Flags: u16 LE, offset 4; APPLICATION_SHARED = 0x0001
- [ ] AppId: u32 LE, offset 6; Windows uses PID (Appendix A <1>)
- [ ] Name: UNICODE_STRING, offset 10; may be empty (Appendix A <10>)
- [ ] Processing: upsert by AppId

### 6.3 OD_APP_REMOVED (§2.2.3.3) — SM→P, 8 bytes

- [ ] HDR.Type=0x0002, HDR.Length=0x0008
- [ ] AppId: u32 LE, offset 4
- [ ] Remove from app list; discard if unknown; also remove all windows for that AppId
      (§3.1.5.3)

### 6.4 OD_WND_CREATED (§2.2.3.4) — SM→P, 16 + cchString*2 bytes

- [ ] HDR.Type=0x0005, HDR.Length=0x0010 + cchString*2
- [ ] Flags: u16 LE, offset 4; WINDOW_SHARED = 0x0001
- [ ] AppId: u32 LE, offset 6 (must reference existing app)
- [ ] WndId: u32 LE, offset 10; Windows uses HWND (Appendix A <3>)
- [ ] Name: UNICODE_STRING, offset 14; window title

### 6.5 OD_WND_REMOVED (§2.2.3.5) — SM→P, 8 bytes

- [ ] HDR.Type=0x0004, HDR.Length=0x0008
- [ ] WndId: u32 LE, offset 4

### 6.6 OD_WND_SHOW (§2.2.3.6) — P→SM, 8 bytes

- [ ] HDR.Type=0x0006, HDR.Length=0x0008
- [ ] WndId: u32 LE, offset 4
- [ ] SM MUST verify MAY_INTERACT before acting (Appendix A <27>)

### 6.7 OD_WND_REGION_UPDATE (§2.2.3.7) — SM→P, 20 bytes

- [ ] HDR.Type=0x000C, HDR.Length=0x0014
- [ ] left/top/right/bottom: 4 × u32 LE at offsets 4/8/12/16
- [ ] Windows NEVER sends this (Appendix A <5>); must parse if received
- [ ] SPEC GAP: no WndId binding — rectangle is advisory only

### 6.8 OD_PARTICIPANT_CREATED (§2.2.4.1) — SM→P, 16 + cchString*2 bytes

- [ ] HDR.Type=0x0008, HDR.Length=0x0010 + cchString*2
- [ ] ParticipantId: u32 LE, offset 4
- [ ] GroupId: u32 LE, offset 8; Windows always 0 (Appendix A <6>)
- [ ] Flags: u16 LE, offset 12
  - [ ] MAY_VIEW       = 0x0001
  - [ ] MAY_INTERACT   = 0x0002
  - [ ] IS_PARTICIPANT = 0x0004 (self-identity unicast; Appendix A <18>)
- [ ] FriendlyName: UNICODE_STRING, offset 14
- [ ] Dual-use: also sent after successful control change to reflect new Flags
- [ ] Windows participants never SEND this PDU (Appendix A <28>)

### 6.9 OD_PARTICIPANT_REMOVED (§2.2.4.2) — SM→P, 16 bytes

- [ ] HDR.Type=0x0007, HDR.Length=0x0010
- [ ] ParticipantId: u32 LE, offset 4
- [ ] DiscType: u32 LE, offset 8 (APP=0x00000000, CLI=0x00000002)
- [ ] DiscCode: u32 LE, offset 12 (HRESULT / Win32)
- [ ] Windows receivers do NOT parse DiscType/DiscCode (Appendix A <21>)

### 6.10 OD_PARTICIPANT_CTRL_CHANGE (§2.2.4.3) — P→SM, 10 bytes

- [ ] HDR.Type=0x0009, HDR.Length=0x000A
- [ ] Flags: u16 LE, offset 4 (Flags BEFORE ParticipantId on wire)
  - [ ] REQUEST_VIEW           = 0x0001
  - [ ] REQUEST_INTERACT       = 0x0002
  - [ ] ALLOW_CONTROL_REQUESTS = 0x0008 (Windows never uses, Appendix A <30>)
- [ ] ParticipantId: u32 LE, offset 6

### 6.11 OD_PARTICIPANT_CTRL_CHANGE_RESPONSE (§2.2.4.4) — SM→P, 14 bytes

- [ ] HDR.Type=0x000D, HDR.Length=0x000E
- [ ] Flags: u16 LE, offset 4 (echoes request)
- [ ] ParticipantId: u32 LE, offset 6
- [ ] ReasonCode: u32 LE, offset 10 (AMBIG: no enum; 0=accept, nonzero=reject)
- [ ] On accept, SM MUST also broadcast OD_PARTICIPANT_CREATED with updated Flags

### 6.12 OD_GRAPHICS_STREAM_PAUSED (§2.2.5.1) — SM→P, 4 bytes

- [ ] HDR.Type=0x000A, HDR.Length=0x0004; header-only

### 6.13 OD_GRAPHICS_STREAM_RESUMED (§2.2.5.2) — SM→P, 4 bytes

- [ ] HDR.Type=0x000B, HDR.Length=0x0004; header-only

## 7. State Machine

### 7.1 Participant (keyed by ParticipantId)

- [ ] UNKNOWN → ACTIVE on OD_PARTICIPANT_CREATED
- [ ] ACTIVE → GONE on OD_PARTICIPANT_REMOVED
- [ ] IS_PARTICIPANT set → store as self_id (persists until channel close)
- [ ] Existing ParticipantId + !IS_PARTICIPANT → replace Flags (control level update)

### 7.2 Application (keyed by AppId)

- [ ] ABSENT → PRESENT on OD_APP_CREATED; duplicate replaces record
- [ ] PRESENT → ABSENT on OD_APP_REMOVED; unknown → silently discard
- [ ] On app removal, cascade-remove all windows with that AppId (§3.1.5.3)

### 7.3 Window (keyed by WndId)

- [ ] ABSENT → PRESENT on OD_WND_CREATED; duplicate replaces record
- [ ] PRESENT → ABSENT on OD_WND_REMOVED; unknown → silently discard

### 7.4 Graphics Stream

- [ ] Initial: RUNNING; → PAUSED on 0x000A; → RUNNING on 0x000B

### 7.5 Filter

- [ ] On state change → flush all stored app/window data (Appendix A <13>)

## 8. Control Request Sequence (§3.3.5.2.3, §3.2.5.2.3)

```
Participant                   Sharing Manager
    |--- CTRL_CHANGE (0x0009) ---->|  Flags=REQUEST_INTERACT|REQUEST_VIEW
    |                              |
    |<-- CTRL_CHANGE_RESPONSE ---- |  (unicast, echo Flags, ReasonCode)
    |    (0x000D)                  |
    |                              |
    |<-- PARTICIPANT_CREATED ------|  (broadcast, updated Flags)
    |    (0x0008)                  |
```

- [ ] Response is unicast to requester (implied by context)
- [ ] Follow-up OD_PARTICIPANT_CREATED broadcast on accept
- [ ] On deny, ReasonCode nonzero, no follow-up broadcast

## 9. Edge Cases

- [ ] Concatenated PDUs: parser uses Length field to advance
- [ ] Oversized Length → terminate connection (Appendix A <8>)
- [ ] Undersized Length → terminate connection (Appendix A <8>)
- [ ] Unknown Type → silently skip
- [ ] Duplicate OD_APP_CREATED / OD_WND_CREATED → replace record
- [ ] Unknown AppId/WndId removes → silently discard
- [ ] cchString = 0 → valid (empty string)
- [ ] cchString > 1024 → protocol violation
- [ ] OD_WND_SHOW without MAY_INTERACT → SM ignores (Appendix A <27>)
- [ ] OD_WND_REGION_UPDATE has no WndId → SPEC GAP, advisory only
- [ ] Multiple OD_PARTICIPANT_CREATED for same id → control-level update
- [ ] Filter reset re-enumeration: batch of Created PDUs follows
- [ ] DoS caps (recommended): MAX_PARTICIPANTS=512, MAX_APPLICATIONS=512, MAX_WINDOWS=1024

## 11. Ambiguities

- [ ] AMBIG-1: OD_WND_REGION_UPDATE has no WndId field — cannot correlate to a window
- [ ] AMBIG-2: ReasonCode has no enum; treat 0=accept, nonzero=reject
- [ ] AMBIG-3: CTRL_CHANGE_RESPONSE unicast/multicast not explicit (assume unicast)
- [ ] AMBIG-4: No single "current controller" PDU; multiple participants may hold MAY_INTERACT
- [ ] AMBIG-5: Downgrade request (REQUEST_VIEW only) handling unspecified
- [ ] AMBIG-6: MS-RDPEPS pre-connection PDU linkage unspecified
