## 2. Variable-Length Integer Encoding

All multi-byte fields are little-endian (§2.2). Variable-length types use a top-bit prefix layout (bit 0 = MSB of first byte).

### 2.1 `TWO_BYTE_UNSIGNED_INTEGER` (§2.2.2.1)

- [ ] Range: `0x0000` – `0x7FFF`
- [ ] First byte layout: `[c(1 bit) | val1(7 bits)]`
  - `c = 0`: 1 byte, value = `val1` (0x00–0x7F)
  - `c = 1`: 2 bytes, value = `(val1 << 8) | val2` (0x0000–0x7FFF)
- [ ] **Boundary**: `0x7F` (1 byte) → `0x80` (2 bytes)
- [ ] Encode: `value <= 0x7F` → `[value as u8]`; else → `[(value >> 8) as u8 | 0x80, value as u8]`
- [ ] Spec example: `0x1A1B` → `[0x9A, 0x1B]`
- [ ] Reject decoded value > `0x7FFF`

### 2.2 `FOUR_BYTE_UNSIGNED_INTEGER` (§2.2.2.3)

- [ ] Range: `0x00000000` – `0x3FFFFFFF`
- [ ] First byte layout: `[c(2 bits) | val1(6 bits)]`
  - `c=0b00`: 1 byte, range 0x00–0x3F
  - `c=0b01`: 2 bytes, range 0x0000–0x3FFF
  - `c=0b10`: 3 bytes, range 0x000000–0x3FFFFF
  - `c=0b11`: 4 bytes, range 0x00000000–0x3FFFFFFF
- [ ] **Boundaries**: 0x3F→0x40, 0x3FFF→0x4000, 0x3FFFFF→0x400000
- [ ] Spec example: `0x001A1B1C` → `[0x9A, 0x1B, 0x1C]` (c=0b10, 3 bytes)
- [ ] Reject > `0x3FFFFFFF`

### 2.3 `EIGHT_BYTE_UNSIGNED_INTEGER` (§2.2.2.5)

- [ ] Range: `0x00` – `0x1FFFFFFFFFFFFFFF`
- [ ] First byte: `[c(3 bits) | val1(5 bits)]`, `c` ∈ 0..7 → 1..8 bytes total
- [ ] **Boundary**: 0x1F→0x20 (1→2 byte transition)
- [ ] Spec example: `0x001A1B1C1D1E1F2A` → `[0xDA, 0x1B, 0x1C, 0x1D, 0x1E, 0x1F, 0x2A]` (7 bytes, c=6)

### 2.4 `TWO_BYTE_SIGNED_INTEGER` (§2.2.2.2)

- [ ] Range: `-0x3FFF` to `+0x3FFF`
- [ ] First byte: `[c(1 bit) | s(1 bit) | val1(6 bits)]`, s=1 negative
- [ ] Spec examples: `-0x1A1B` → `[0xDA, 0x1B]`; `-0x0002` → `[0x42]`

### 2.5 `FOUR_BYTE_SIGNED_INTEGER` (§2.2.2.4)

- [ ] Range: `-0x1FFFFFFF` to `+0x1FFFFFFF`
- [ ] First byte: `[c(2 bits) | s(1 bit) | val1(5 bits)]`
- [ ] Spec examples: `-0x001A1B1C` → `[0xBA, 0x1B, 0x1C]`; `-0x00000002` → `[0x22]`

## 3. PDU Wire Formats

### 3.1 `RDPINPUT_SC_READY_PDU` (§2.2.3.1) — Server → Client

| Field              | Type              | Offset | Size | Notes |
|--------------------|-------------------|--------|------|-------|
| `header`           | RDPINPUT_HEADER   | 0      | 6    | eventId = `0x0001` |
| `protocolVersion`  | u32 LE            | 6      | 4    | see constants |
| `supportedFeatures`| u32 LE            | 10     | 4    | **optional**, V300 only |

- [ ] `pduLength` = 10 (V100/V101/V200) or 14 (V300)
- [ ] Determine presence of `supportedFeatures` by **`pduLength == 14`**, NOT by `protocolVersion` alone (§2.2.3.1 uses SHOULD, not MUST)
- [ ] If `pduLength == 10`, treat `supportedFeatures` as absent (0)

### Protocol Version Constants

- [ ] `RDPINPUT_PROTOCOL_V100` = `0x00010000`
- [ ] `RDPINPUT_PROTOCOL_V101` = `0x00010001`
- [ ] `RDPINPUT_PROTOCOL_V200` = `0x00020000` (+ pen)
- [ ] `RDPINPUT_PROTOCOL_V300` = `0x00030000` (+ granular features)

### `supportedFeatures` Flags (SC_READY only)

- [ ] `SC_READY_MULTIPEN_INJECTION_SUPPORTED` = `0x00000001`

---

### 3.2 `RDPINPUT_CS_READY_PDU` (§2.2.3.2) — Client → Server

| Field             | Type            | Offset | Size | Notes |
|-------------------|-----------------|--------|------|-------|
| `header`          | RDPINPUT_HEADER | 0      | 6    | eventId = `0x0002` |
| `flags`           | u32 LE          | 6      | 4    | see below |
| `protocolVersion` | u32 LE          | 10     | 4    | |
| `maxTouchContacts`| u16 LE          | 14     | 2    | |

- [ ] `pduLength` = **16** (fixed)
- [ ] `flags` is **4 bytes** (not 2)

### `flags` Constants

- [ ] `CS_READY_FLAGS_SHOW_TOUCH_VISUALS` = `0x00000001`
- [ ] `CS_READY_FLAGS_DISABLE_TIMESTAMP_INJECTION` = `0x00000002` (SHOULD NOT send to V100 server)
- [ ] `CS_READY_FLAGS_ENABLE_MULTIPEN_INJECTION` = `0x00000004`

### `maxTouchContacts`

- [ ] u16 field (spec has no explicit upper bound)
- [ ] Windows default: 10; roadmap target: 256
- [ ] **Ambiguity**: spec silent on upper limit; implementation policy decides

---

### 3.3 `RDPINPUT_TOUCH_EVENT_PDU` (§2.2.3.3) — Client → Server

| Field         | Type                          | Notes |
|---------------|-------------------------------|-------|
| `header`      | RDPINPUT_HEADER               | eventId = `0x0003` |
| `encodeTime`  | FOUR_BYTE_UNSIGNED_INTEGER    | ms from oldest frame to encode time |
| `frameCount`  | TWO_BYTE_UNSIGNED_INTEGER     | number of frames |
| `frames`      | array of RDPINPUT_TOUCH_FRAME | oldest → newest |

- [ ] `frameCount` range 0..0x7FFF (spec limit is TWO_BYTE max; roadmap 256 = policy)
- [ ] `encodeTime` still valid when `DISABLE_TIMESTAMP_INJECTION` set (server ignores)
- [ ] Total `pduLength` = 6 + sizeof(encodeTime) + sizeof(frameCount) + Σ frame sizes

---

### 3.4 `RDPINPUT_TOUCH_FRAME` (§2.2.3.3.1)

| Field          | Type                          | Notes |
|----------------|-------------------------------|-------|
| `contactCount` | TWO_BYTE_UNSIGNED_INTEGER     | 0..0x7FFF |
| `frameOffset`  | EIGHT_BYTE_UNSIGNED_INTEGER   | microseconds since previous frame; 0 for first |
| `contacts`     | array of RDPINPUT_TOUCH_CONTACT | length = contactCount |

- [ ] First frame in PDU: `frameOffset = 0`
- [ ] Server ignores `frameOffset` when `DISABLE_TIMESTAMP_INJECTION` was set

### 3.5 `RDPINPUT_TOUCH_CONTACT` (§2.2.3.3.1.1)

> **Spec name is `RDPINPUT_TOUCH_CONTACT`**, not `RDPINPUT_CONTACT_DATA` as written in the roadmap.

| Field                | Type                         | Notes |
|----------------------|------------------------------|-------|
| `contactId`          | u8                           | fixed 1 byte, 0x00–0xFF |
| `fieldsPresent`      | TWO_BYTE_UNSIGNED_INTEGER    | optional-field presence |
| `x`                  | FOUR_BYTE_SIGNED_INTEGER     | virtual-desktop coords |
| `y`                  | FOUR_BYTE_SIGNED_INTEGER     | |
| `contactFlags`       | FOUR_BYTE_UNSIGNED_INTEGER   | state flags (see combos) |
| `contactRectLeft`    | TWO_BYTE_SIGNED_INTEGER      | optional (CONTACTRECT) |
| `contactRectTop`     | TWO_BYTE_SIGNED_INTEGER      | optional (CONTACTRECT) |
| `contactRectRight`   | TWO_BYTE_SIGNED_INTEGER      | optional (CONTACTRECT) |
| `contactRectBottom`  | TWO_BYTE_SIGNED_INTEGER      | optional (CONTACTRECT) |
| `orientation`        | FOUR_BYTE_UNSIGNED_INTEGER   | optional (ORIENTATION) |
| `pressure`           | FOUR_BYTE_UNSIGNED_INTEGER   | optional (PRESSURE) |

### `fieldsPresent` Flags

- [ ] `TOUCH_CONTACT_CONTACTRECT_PRESENT` = `0x0001` — all 4 rect fields together
- [ ] `TOUCH_CONTACT_ORIENTATION_PRESENT` = `0x0002`
- [ ] `TOUCH_CONTACT_PRESSURE_PRESENT` = `0x0004`

### `contactFlags` Bits

- [ ] `CONTACT_FLAG_DOWN` = `0x0001`
- [ ] `CONTACT_FLAG_UPDATE` = `0x0002`
- [ ] `CONTACT_FLAG_UP` = `0x0004`
- [ ] `CONTACT_FLAG_INRANGE` = `0x0008`
- [ ] `CONTACT_FLAG_INCONTACT` = `0x0010`
- [ ] `CONTACT_FLAG_CANCELED` = `0x0020`

### Valid `contactFlags` Combinations (exactly 8)

| Combination | Hex | Meaning |
|---|---|---|
| `UP` | 0x04 | contact left, out of range |
| `UP \| CANCELED` | 0x24 | canceled while lifting |
| `UPDATE` | 0x02 | position update out of range |
| `UPDATE \| CANCELED` | 0x22 | position update, canceled |
| `DOWN \| INRANGE \| INCONTACT` | 0x19 | new contact on surface |
| `UPDATE \| INRANGE \| INCONTACT` | 0x1A | position update while touching |
| `UP \| INRANGE` | 0x0C | lifted but still hovering |
| `UPDATE \| INRANGE` | 0x0A | position update while hovering |

- [ ] Encoder validates one of these 8 combos
- [ ] Decoder rejects invalid combos

### `orientation` / `pressure` Constraints

- [ ] `orientation` range: 0–359 (`0x000`–`0x167`), 0=y-axis up, increases CCW
- [ ] `pressure` range: 0–1024 (`0x000`–`0x400`)

## 4. Protocol Version Negotiation

- [ ] Server sends SC_READY with its max version
- [ ] Client responds with CS_READY; negotiated version = `min(server, client)`
- [ ] V300 SC_READY: read `supportedFeatures` iff `pduLength == 14`
- [ ] `DISABLE_TIMESTAMP_INJECTION` SHOULD NOT be sent when server advertised V100
- [ ] **Ambiguity**: client requesting multipen when server didn't advertise it — behavior unspecified

## 5. Contact State Machine (§3.1.1.1)

```
Out of Range ──[DOWN|INRANGE|INCONTACT]──> Engaged
Engaged      ──[UP]──────────────────────> Out of Range
Engaged      ──[UP|INRANGE]──────────────> Hovering
Hovering     ──[DOWN|INRANGE|INCONTACT]──> Engaged
Hovering     ──[DISMISS_HOVERING]────────> Out of Range
```

- [ ] Three states: `OutOfRange`, `Hovering`, `Engaged`
- [ ] `UPDATE|INRANGE|INCONTACT` valid only from Engaged
- [ ] `UPDATE|INRANGE` valid only from Hovering
- [ ] Position MUST NOT change on engaged→hovering or engaged→out-of-range transition (§3.1.1.1)

## 6. DVC Processor State Machine

```
WaitScReady
  └── on SC_READY (0x0001): store version, supportedFeatures → SendCsReady
SendCsReady
  └── immediately send CS_READY → Ready
Ready
  ├── on SUSPEND_INPUT (0x0004): InputTransmissionSuspended = true
  ├── on RESUME_INPUT (0x0005): InputTransmissionSuspended = false
  ├── API send_touch_frame: if !suspended → send TOUCH_EVENT_PDU
  ├── API dismiss_hovering(id): send DISMISS_HOVERING PDU
  └── unknown eventIds: ignore (§3.1.5.1)
```

- [ ] `InputTransmissionSuspended` ADM (§3.3.1.1): starts `false`
- [ ] When true: MUST NOT send TOUCH_EVENT_PDU
- [ ] Double SUSPEND: ignored (§3.3.5.4)
- [ ] On channel close: reset state

## 7. Message Flow

```
Server                                    Client
  │                                          │
  │──── SC_READY (0x0001) ───────────────────▶│  server speaks first
  │                                          │
  │◀─── CS_READY (0x0002) ───────────────────│
  │                                          │
  │◀─── TOUCH_EVENT_PDU (0x0003) ────────────│  0 or more
  │                                          │
  │──── SUSPEND_INPUT (0x0004) ──────────────▶│  server-initiated
  │──── RESUME_INPUT  (0x0005) ──────────────▶│
  │                                          │
  │◀─── DISMISS_HOVERING (0x0006) ───────────│  client-initiated
```

- [ ] Server MUST send SC_READY before client sends CS_READY
- [ ] Client MUST NOT send touch frames before SC_READY
- [ ] Second SC_READY (reconnect) → client re-sends CS_READY

### Spec Ambiguities / Gaps

- [ ] **Max contacts per frame**: no MUST; `contactCount` field is TWO_BYTE max. Roadmap's 256 is policy.
- [ ] **Max frames per PDU**: no MUST; `frameCount` field is TWO_BYTE max. Roadmap's 256 is policy.
- [ ] **Multipen negotiation mismatch**: client requests multipen without server advertisement → unspecified.
- [ ] **V300 `supportedFeatures`**: spec uses SHOULD. Decide presence by `pduLength`, not `protocolVersion`.
