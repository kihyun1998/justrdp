## 1. Transport (MS-RDPEDC В§2.1)

- [ ] MS-RDPEDC does NOT use a Dynamic Virtual Channel. Orders are sent as GDI **Alternate Secondary Orders** embedded in the normal RDP Update PDU stream.
- [ ] Every PDU begins with a 1-byte `header` that is an Alternate Secondary Order Header (MS-RDPEGDI В§2.2.2.2.1.3.1.1).
- [ ] The `header` byte encodes two sub-fields:
  - bits 0вЂ“5: `orderType` вЂ” MUST be `TS_ALTSEC_COMPDESK_FIRST` = `0x0C` (12) for all MS-RDPEDC orders
  - bits 6вЂ“7: `controlFlags` вЂ” MUST be `0x2` (`TS_SECONDARY`, i.e. Alternate Secondary Order)
  - Resulting constant header byte value: `(0x0C << 2) | 0x2` = `0x32`
- [ ] The roadmap's "requires: 7.3 DVC framework" is a documentation error for this spec; MS-RDPEDC requires only MS-RDPBCGR + MS-RDPEGDI alternate secondary order transport.
- [ ] Capability prerequisite: both client and server MUST include `TS_COMPDESK_CAPABILITYSET` (MS-RDPBCGR В§2.2.7.2.8) in their Active/Confirm Active PDU with `CompDeskSupportLevel = COMPDESK_SUPPORTED (0x0001)`.

## 3. Wire Format вЂ” Operation Codes

All MS-RDPEDC PDUs share the same 4-byte common prefix. Define these constants (MS-RDPEDC В§2.2):

- [ ] `TS_ALTSEC_COMPDESK_FIRST = 0x0C` вЂ” orderType embedded in header byte (MS-RDPEGDI В§2.2.2.2.1.3.1.1)
- [ ] `COMPDESKTOGGLE = 0x01` вЂ” operation code for TS_COMPDESK_TOGGLE (MS-RDPEDC В§2.2.1.1)
- [ ] `LSURFACE_CREATE_DESTROY = 0x02` вЂ” operation code for TS_COMPDESK_LSURFACE (MS-RDPEDC В§2.2.2.1)
- [ ] `SURFOBJ_CREATE_DESTROY = 0x03` вЂ” operation code for TS_COMPDESK_SURFOBJ (MS-RDPEDC В§2.2.2.2)
- [ ] `REDIRSURF_ASSOC_DEASSOC_LSURFACE = 0x04` вЂ” operation code for TS_COMPDESK_REDIRSURF_ASSOC_LSURFACE (MS-RDPEDC В§2.2.2.3)
- [ ] `LSURFACE_COMPREF_PENDING = 0x05` вЂ” operation code for TS_COMPDESK_LSURFACE_COMPREF_PENDING (MS-RDPEDC В§2.2.2.4)
- [ ] `SURFOBJSWITCH = 0x06` вЂ” operation code for TS_COMPDESK_SWITCH_SURFOBJ (MS-RDPEDC В§2.2.3.1)
- [ ] `FLUSHCOMPOSEONCE = 0x07` вЂ” operation code for TS_COMPDESK_FLUSH_COMPOSEONCE (MS-RDPEDC В§2.2.3.2)

### Common Header Prefix (4 bytes, every PDU)

| Offset | Size | Type  | Field       | Notes |
|-------:|-----:|-------|-------------|-------|
| 0      | 1    | u8    | `header`    | Alternate Secondary Order Header; MUST be `0x32` (orderType=0x0C, controlFlags=0x2) |
| 1      | 1    | u8    | `operation` | PDU-specific operation code (values listed above) |
| 2      | 2    | u16LE | `size`      | Byte count of PDU body **following** this `size` field (i.e. total PDU size в€’ 4) |

## 4. Wire Format вЂ” PDU 1: TS_COMPDESK_TOGGLE (MS-RDPEDC В§2.2.1.1)

Direction: **server в†’ client**. Signals composition mode or desktop mode changes.

Total PDU size: **5 bytes**.

| Offset | Size | Type  | Field       | Constant value |
|-------:|-----:|-------|-------------|----------------|
| 0      | 1    | u8    | `header`    | `0x32` |
| 1      | 1    | u8    | `operation` | `COMPDESKTOGGLE (0x01)` |
| 2      | 2    | u16LE | `size`      | `0x0001` (1 byte follows) |
| 4      | 1    | u8    | `eventType` | See table below |

### eventType values (MS-RDPEDC В§2.2.1.1)

- [ ] `REDIRMODE_COMPOSITION_OFF = 0x00` вЂ” server leaving composition mode
- [ ] `REDIRMODE_RESERVED_00 = 0x01` вЂ” reserved; SHOULD be ignored if received
- [ ] `REDIRMODE_RESERVED_01 = 0x02` вЂ” reserved; SHOULD be ignored if received
- [ ] `REDIRMODE_COMPOSITION_ON = 0x03` вЂ” server entering composition mode
- [ ] `REDIRMODE_DWM_DESK_ENTER = 0x04` вЂ” switch from non-composed to composed desktop
- [ ] `REDIRMODE_DWM_DESK_LEAVE = 0x05` вЂ” switch from composed to non-composed desktop

## 5. Wire Format вЂ” PDU 2: TS_COMPDESK_LSURFACE (MS-RDPEDC В§2.2.2.1)

Direction: **server в†’ client**. Creates or destroys a logical surface.

Total PDU size: **38 bytes** (`size` field = `0x0022` = 34 bytes after the `size` field; 4 header bytes + 34 body bytes = 38 total).

| Offset | Size | Type  | Field       | Notes |
|-------:|-----:|-------|-------------|-------|
| 0      | 1    | u8    | `header`    | `0x32` |
| 1      | 1    | u8    | `operation` | `LSURFACE_CREATE_DESTROY (0x02)` |
| 2      | 2    | u16LE | `size`      | `0x0022` (34) |
| 4      | 1    | u8    | `fCreate`   | `0x01` = create, `0x00` = destroy |
| 5      | 1    | u8    | `flags`     | Bitfield (valid only when `fCreate=1`); see below |
| 6      | 8    | u64LE | `hLsurface` | Surface-manager-generated logical surface identifier |
| 14     | 4    | u32LE | `width`     | MUST be `0x00000000`; ignored by client |
| 18     | 4    | u32LE | `height`    | MUST be `0x00000000`; ignored by client |
| 22     | 8    | u64LE | `hwnd`      | Window handle on server associated with logical surface |
| 30     | 8    | u64LE | `luid`      | MUST be `0x0000000000000000`; ignored by client |

### `flags` bit values (MS-RDPEDC В§2.2.2.1)

- [ ] `TS_COMPDESK_HLSURF_COMPOSEONCE = 0x01` вЂ” logical surface is a compose-once surface
- [ ] `TS_COMPDESK_HLSURF_REDIRECTION = 0x04` вЂ” logical surface is a redirection surface
- [ ] When `fCreate = 0x00`, `flags` field MUST be ignored on decode

## 6. Wire Format вЂ” PDU 3: TS_COMPDESK_SURFOBJ (MS-RDPEDC В§2.2.2.2)

Direction: **server в†’ client**. Creates or destroys a redirection surface.

Total PDU size: **26 bytes** (`size` field = `0x0016` = 22 bytes; 4 + 22 = 26 total).

| Offset | Size | Type  | Field        | Notes |
|-------:|-----:|-------|--------------|-------|
| 0      | 1    | u8    | `header`     | `0x32` |
| 1      | 1    | u8    | `operation`  | `SURFOBJ_CREATE_DESTROY (0x03)` |
| 2      | 2    | u16LE | `size`       | `0x0016` (22) |

### `cacheId` bit-field semantics (MS-RDPEDC В§2.2.2.2)

- [ ] Bits 0вЂ“30: surface identifier (31 bits)
- [ ] Bit 31 (`0x80000000`): `0` = create event, `1` = destroy event

## 11. State Machine (MS-RDPEDC В§3.2.5 Client)

The client tracks two independent state variables and one lookup table per type:

### Composition Mode (MS-RDPEDC В§3.2.5.1.1)

```
COMPOSITION_OFF  в”Ђв”ЂCOMPOSITION_ONв”Ђв”Ђв–є  COMPOSITION_ON
COMPOSITION_ON   в”Ђв”ЂCOMPOSITION_OFFв”Ђв”Ђв–є COMPOSITION_OFF
COMPOSITION_OFF  в”Ђ(any event)в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–є  stay in COMPOSITION_OFF, ignore
```

- [ ] Initial state: `COMPOSITION_OFF`
- [ ] On `REDIRMODE_COMPOSITION_ON (0x03)`: enter `COMPOSITION_ON`; allocate surface tables
- [ ] On `REDIRMODE_COMPOSITION_OFF (0x00)`: enter `COMPOSITION_OFF`; release all surface state
- [ ] On `REDIRMODE_DWM_DESK_ENTER (0x04)`: switch to "composed desktop" sub-mode while already `COMPOSITION_ON`
- [ ] On `REDIRMODE_DWM_DESK_LEAVE (0x05)`: switch to "non-composed desktop" sub-mode while `COMPOSITION_ON`
- [ ] Duplicated or out-of-order TOGGLE messages MUST be silently ignored (MS-RDPEDC В§2.2.1 note)
- [ ] Reserved values `0x01` and `0x02` SHOULD be ignored if received

### Logical Surface Table (MS-RDPEDC В§3.2.5.2.1)

- [ ] State: `BTreeMap<u64 /*hLSurface*/, LogicalSurfaceEntry>` (no_std friendly)
- [ ] `LogicalSurfaceEntry` fields: `flags: u8`, `hwnd: u64`, `compref_pending: bool`
- [ ] On `TS_COMPDESK_LSURFACE` with `fCreate=1`: insert entry
- [ ] On `TS_COMPDESK_LSURFACE` with `fCreate=0`: remove entry
- [ ] On `TS_COMPDESK_LSURFACE_COMPREF_PENDING`: set `compref_pending = true` for named entry; do not release until compositor retrieves
- [ ] On unknown `hLSurface` in COMPREF_PENDING: implementation-defined (log and ignore)

### Redirection Surface Table (MS-RDPEDC В§3.2.5.2.2)

- [ ] State: `BTreeMap<u32 /*cacheId without bit-31*/, RedirSurfaceEntry>`
- [ ] `RedirSurfaceEntry` fields: `surface_bpp: u8`, `h_surf: u64`, `cx: u32`, `cy: u32`, `assoc_lsurface: Option<u64>`
- [ ] On `TS_COMPDESK_SURFOBJ` with `cacheId bit31=0` (create): insert entry with `cacheId & 0x7FFFFFFF`
- [ ] On `TS_COMPDESK_SURFOBJ` with `cacheId bit31=1` (destroy): remove entry with `cacheId & 0x7FFFFFFF`
- [ ] On `TS_COMPDESK_REDIRSURF_ASSOC_LSURFACE` with `fAssociate=1`: set `assoc_lsurface` in surface entry
- [ ] On `TS_COMPDESK_REDIRSURF_ASSOC_LSURFACE` with `fAssociate=0`: clear `assoc_lsurface`

### Drawing Retarget (MS-RDPEDC В§3.2.5.3)

- [ ] `TS_COMPDESK_SWITCH_SURFOBJ`: subsequent drawing orders target the surface identified by `cacheId` until next SWITCH or FLUSH
- [ ] `TS_COMPDESK_FLUSH_COMPOSEONCE`: notify that the compose-once drawing cycle is complete; MAY trigger compositor pass

## 12. Edge Cases

- [ ] `fCreate = 0` on a `hLSurface` not in the table: SHOULD be ignored (duplicate destroy)
- [ ] `cacheId` destroy for unknown surface: SHOULD be ignored
- [ ] `fAssociate` on unknown `hLSurface` or unknown `cacheId`: SHOULD be ignored
- [ ] `eventType` values 0x01 and 0x02 in TS_COMPDESK_TOGGLE: explicitly reserved; SHOULD be ignored (MS-RDPEDC В§2.2.1.1)
- [ ] `eventType` values beyond 0x05: no spec guidance вЂ” treat as unknown, log and ignore
- [ ] `surfaceBpp` in TS_COMPDESK_SURFOBJ: spec does not constrain the value; implementation may want to validate against known values (8, 16, 24, 32)
- [ ] `width` and `height` fields in TS_COMPDESK_LSURFACE are explicitly unused (MUST be 0 on send, ignored on receive); decode MUST NOT use them
- [ ] `luid` field in TS_COMPDESK_LSURFACE: MUST be 0 on send, ignored on receive
- [ ] `flags` field in TS_COMPDESK_SURFOBJ: MUST be 0 on send, MUST be ignored on receive
- [ ] `FLUSHCOMPOSEONCE` on a surface not flagged `TS_COMPDESK_HLSURF_COMPOSEONCE`: SHOULD NOT be received per spec (MS-RDPEDC В§3.2.5.3.2 / В§2.2.3.2 note); log and ignore
- [ ] PDU with unknown `operation` byte: unknown PDU; decode the `size` field and skip `size` bytes (forward-compat)
- [ ] `size` field mismatch (decoded bytes != expected for known operation): SHOULD return a decode error
- [ ] Max table size: spec does not define a limit; implementation MUST bound BTreeMap growth (recommend a `MAX_LOGICAL_SURFACES` and `MAX_REDIR_SURFACES` guard)

## 15. Spec Ambiguities / Flags

- [ ] AMBIGUITY: The spec says "the `size` field is the size of the order data that follows the size field" but does not explicitly state little-endian for this field. All other multi-byte fields in RDP are LE; assume LE.
- [ ] AMBIGUITY: `TS_COMPDESK_LSURFACE.flags` is a u8 bitfield but only 2 bits are defined (`0x01` and `0x04`). Undefined bits SHOULD be zero; client decode SHOULD mask to known bits only.
- [ ] AMBIGUITY: Transport section in spec (В§2.1) is empty in the live HTML rendering (word_count=0). The mechanism is inferred from: (a) every PDU uses an Alternate Secondary Order header identical in format to MS-RDPEGDI В§2.2.2.2.1.3.1.1, and (b) the В§1.3.2 overview says "Relationship to Update Orders PDU". The ordering is: these PDUs ride inside standard RDP Update PDUs, multiplexed with normal drawing orders вЂ” not over a separate channel.
- [ ] NOTE: MS-RDPCR2 (Composited Remoting V2, 136 messages) is a completely different protocol. The roadmap В§9.12 "MS-RDPECR2" is a typo for MS-RDPEDC. MS-RDPCR2 would be an entirely separate roadmap entry.
