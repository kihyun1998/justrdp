## 1. Transport / Channel Layout (§2.1, §1.3.1.1)

- [ ] **Control DVC channel name**: `"URBDRC"` — ANSI encoded, NUL-terminated string (§2.1)
  - This is the **single control channel** carrying capability exchange + Device Sink interface
- [ ] **Per-device DVC naming**: NOT spec-named. Each redirected USB device gets its own fresh DVC, created by the server in response to `ADD_VIRTUAL_CHANNEL`. The server picks the channel name (conventionally "URBDRC" also; distinguishing is via ChannelId, not name). Implementer MUST NOT hard-code the per-device name — accept whatever name the server's `CREATE_REQUEST_PDU` supplies.
- [ ] Protocol is terminated by closing the underlying DVC (§1.5)
- [ ] Malformed or out-of-sequence packets MUST be ignored, not rejected as errors (§3.1.5)
- [ ] Control channel carries: RIM_EXCHANGE_CAPABILITY, CHANNEL_CREATED, ADD_VIRTUAL_CHANNEL
- [ ] Per-device channel carries: ADD_DEVICE (first message), CHANNEL_CREATED, REGISTER_REQUEST_CALLBACK, IO_CONTROL/INTERNAL_IO_CONTROL/QUERY_DEVICE_TEXT/TRANSFER_*_REQUEST/CANCEL_REQUEST/RETRACT_DEVICE, and all completions

## 2. SHARED_MSG_HEADER (§2.2.1)

**All multi-byte fields are little-endian throughout MS-RDPEUSB unless explicitly noted.**

### 2.1 Wire layout

| Field         | Width                 | Offset | Type                   | Notes                                      |
|---------------|-----------------------|--------|------------------------|--------------------------------------------|
| `InterfaceId` | **30 bits** (low)     | 0      | u32 LE (bitfield)      | Low 30 bits of the first 4-byte word       |
| `Mask`        | **2 bits** (high)     | 0      | u32 LE (bitfield)      | High 2 bits of the same 4-byte word        |
| `MessageId`   | 4 bytes               | 4      | u32 LE unsigned        |                                            |
| `FunctionId`  | 4 bytes               | 8      | u32 LE unsigned        | Present on all packets except **responses**|
| `messagePayload` | variable           | 8 or 12| bytes                  | Body, interface-dependent                  |

- [ ] First 32-bit word layout (little-endian, **byte 0 LSB**): bits [0..30] = `InterfaceId`, bits [30..32] = `Mask`
  - Encode: `word0 = (InterfaceId & 0x3FFF_FFFF) | ((Mask & 0x3) << 30)`
  - Decode: `InterfaceId = word0 & 0x3FFF_FFFF`, `Mask = (word0 >> 30) & 0x3`
- [ ] **`Mask` values** (§2.2.1):
  - [ ] `STREAM_ID_NONE = 0x0` — capability exchange ONLY (§2.2.3); MUST NOT appear elsewhere
  - [ ] `STREAM_ID_PROXY = 0x1` — non-response (request / notification)
  - [ ] `STREAM_ID_STUB = 0x2` — response message
- [ ] **`FunctionId` is OMITTED on responses**. A response is identified by `Mask == STREAM_ID_STUB`. Responses: `RIM_EXCHANGE_CAPABILITY_RESPONSE`, `QUERY_DEVICE_TEXT_RSP`. All completion messages on the Request Completion Interface also OMIT FunctionId per §2.2.7.1/2/3... WAIT: re-check. §2.2.7.1 says "Mask = STREAM_ID_PROXY, FunctionId = IOCONTROL_COMPLETION (0x00000100)" — so completions DO include FunctionId. Only RIM_EXCHANGE_CAPABILITY_RESPONSE and QUERY_DEVICE_TEXT_RSP use STREAM_ID_STUB.
- [ ] **Header size**:
  - Request / notification (FunctionId present): **12 bytes**
  - Response (STREAM_ID_STUB, FunctionId absent): **8 bytes**
- [ ] InterfaceId lifetime: valid until sent/received in `IFACE_RELEASE` (MS-RDPEXPS §2.2.2.2); after that the ID is invalid
- [ ] Top 2 bits of `NetInterfaceId` from a QI_RSP (MS-RDPEXPS §2.2.2.1.2) MUST be ignored when adopted as InterfaceId

### 2.2 Common FunctionId values (all interfaces)

- [ ] `RIMCALL_RELEASE = 0x00000001` — release an interface ID
- [ ] `RIMCALL_QUERYINTERFACE = 0x00000002` — query for a new interface

## 3. Interface IDs (§2.2.1, §2.2.3–§2.2.7)

Default / well-known InterfaceId values (30-bit field):

| InterfaceId  | Purpose                                                                 | Direction              |
|--------------|-------------------------------------------------------------------------|------------------------|
| `0x00000000` | Capabilities Negotiator (§2.2.3) — RIM_EXCHANGE_CAPABILITY              | S→C request, C→S resp  |
| `0x00000001` | Device Sink Interface (§2.2.4) — ADD_VIRTUAL_CHANNEL, ADD_DEVICE        | Client → Server        |
| `0x00000002` | Channel Notification, **server→client** CHANNEL_CREATED sender (§2.2.5) | Server → Client        |
| `0x00000003` | Channel Notification, **client→server** CHANNEL_CREATED sender (§2.2.5) | Client → Server        |
| `UsbDevice`  | USB Device Interface per-device ID (§2.2.6) — IO_CONTROL, TRANSFER_*, CANCEL, RETRACT, REGISTER_REQUEST_CALLBACK | Server → Client |
| `RequestCompletion` | Request Completion Interface per-device ID (§2.2.7) — IOCONTROL_COMPLETION, URB_COMPLETION, URB_COMPLETION_NO_DATA | Client → Server |

- [ ] The `UsbDevice` interface ID is ALLOCATED BY THE CLIENT in `ADD_DEVICE.UsbDevice` (§2.2.4.2)
- [ ] The `RequestCompletion` interface ID is ALLOCATED BY THE SERVER in `REGISTER_REQUEST_CALLBACK.RequestCompletion` (§2.2.6.2)
- [ ] Client MUST track the mapping `UsbDevice → RequestCompletion` per device
- [ ] Duplicate UsbDevice interface IDs at the server → server ignores new ADD_DEVICE, keeps original (§3.2.5.1.2)

### 3.1 FunctionId values per interface

**Capabilities Negotiator (InterfaceId = 0x00000000):**
- [ ] `RIM_EXCHANGE_CAPABILITY_REQUEST = 0x00000100` (server → client)

**Device Sink Interface (InterfaceId = 0x00000001):**
- [ ] `ADD_VIRTUAL_CHANNEL = 0x00000100` (client → server)
- [ ] `ADD_DEVICE = 0x00000101` (client → server)

**Channel Notification Interface (InterfaceId = 0x00000002 or 0x00000003):**
- [ ] `CHANNEL_CREATED = 0x00000100` (both directions)

**Server USB Device Interface (InterfaceId = ADD_DEVICE.UsbDevice):**
- [ ] `CANCEL_REQUEST = 0x00000100`
- [ ] `REGISTER_REQUEST_CALLBACK = 0x00000101`
- [ ] `IO_CONTROL = 0x00000102`
- [ ] `INTERNAL_IO_CONTROL = 0x00000103`
- [ ] `QUERY_DEVICE_TEXT = 0x00000104`
- [ ] `TRANSFER_IN_REQUEST = 0x00000105`
- [ ] `TRANSFER_OUT_REQUEST = 0x00000106`
- [ ] `RETRACT_DEVICE = 0x00000107`

**Client Request Completion Interface (InterfaceId = REGISTER_REQUEST_CALLBACK.RequestCompletion):**
- [ ] `IOCONTROL_COMPLETION = 0x00000100`
- [ ] `URB_COMPLETION = 0x00000101`
- [ ] `URB_COMPLETION_NO_DATA = 0x00000102`

## 4. Capability Exchange (§2.2.3)

### 4.1 `RIM_EXCHANGE_CAPABILITY_REQUEST` (§2.2.3.1) — server → client

| Field             | Width   | Offset | Value / Notes                                  |
|-------------------|---------|--------|------------------------------------------------|
| `Header`          | 12 B    | 0      | InterfaceId=0x00000000, Mask=STREAM_ID_NONE(0), FunctionId=0x00000100 |
| `CapabilityValue` | 4 B LE  | 12     | u32, MUST contain `RIM_CAPABILITY_VERSION_01`  |

- [ ] Total size: **16 bytes**
- [ ] `RIM_CAPABILITY_VERSION_01 = 0x00000001` — MUST be present (§2.2.3.1 table)
- [ ] Header Mask MUST be `STREAM_ID_NONE` (§2.2.3.1)
- [ ] On receipt the client MUST reply with RIM_EXCHANGE_CAPABILITY_RESPONSE (§3.3.5.5.1)

### 4.2 `RIM_EXCHANGE_CAPABILITY_RESPONSE` (§2.2.3.2) — client → server

| Field             | Width   | Offset | Value / Notes                                  |
|-------------------|---------|--------|------------------------------------------------|
| `Header`          | **8 B** | 0      | InterfaceId=0x00000000, Mask=STREAM_ID_NONE(0), **no FunctionId** |
| `CapabilityValue` | 4 B LE  | 8      | u32, MUST contain `RIM_CAPABILITY_VERSION_01` (0x00000001) |
| `Result`          | 4 B LE  | 12     | u32, HRESULT of operation                      |

- [ ] Total size: **16 bytes**
- [ ] Header InterfaceId and MessageId SHOULD equal those of the matching REQUEST
- [ ] Mask MUST be STREAM_ID_NONE (note: this is the one response message that uses STREAM_ID_NONE rather than STREAM_ID_STUB — §2.2.3.2)
- [ ] On success `Result = S_OK (0x00000000)`
- [ ] Send ordering: server MUST send RIM_EXCHANGE_CAPABILITY_REQUEST **before** CHANNEL_CREATED (§3.2.5.5.1)

### 4.3 `CHANNEL_CREATED` (§2.2.5.1) — both directions

| Field          | Width  | Offset | Value / Notes                                  |
|----------------|--------|--------|------------------------------------------------|
| `Header`       | 12 B   | 0      | InterfaceId=0x2 (S→C) or 0x3 (C→S), Mask=STREAM_ID_PROXY, FunctionId=0x00000100 |
| `MajorVersion` | 4 B LE | 12     | u32, MUST be `1`                               |
| `MinorVersion` | 4 B LE | 16     | u32, MUST be `0`                               |
| `Capabilities` | 4 B LE | 20     | u32, MUST be `0`                               |

- [ ] Total size: **24 bytes**
- [ ] Server sends with InterfaceId = `0x00000002`
- [ ] Client sends with InterfaceId = `0x00000003`
- [ ] Version mismatch → the receiving side MUST close the DVC (§3.2.5.2.2 / §3.3.5.2.2)
- [ ] Channel-setup ordering (§1.3.1.1, §3.2.5.5.1):
  1. Server sends `RIM_EXCHANGE_CAPABILITY_REQUEST`
  2. Client sends `RIM_EXCHANGE_CAPABILITY_RESPONSE`
  3. Server sends `CHANNEL_CREATED`
  4. Client sends `CHANNEL_CREATED`
  After step 4, protocol enters Ready state (§3.1)

## 5. Device Enumeration — Device Sink Interface (§2.2.4)

### 5.1 `ADD_VIRTUAL_CHANNEL` (§2.2.4.1) — client → server

| Field    | Width | Offset | Value                                                |
|----------|-------|--------|------------------------------------------------------|
| `Header` | 12 B  | 0      | InterfaceId=0x00000001, Mask=STREAM_ID_PROXY, FunctionId=0x00000100 |

- [ ] Total size: **12 bytes**
- [ ] No payload. Purely a trigger; server responds by opening a new DVC for the upcoming device (§3.3.5.1.1, §3.2.5.1.1)
- [ ] Invalid ADD_VIRTUAL_CHANNEL → server MUST terminate the DVC (§3.2.5.1.1)

### 5.2 `ADD_DEVICE` (§2.2.4.2) — client → server (sent on the NEW per-device DVC)

| Field                   | Width        | Offset  | Value / Notes                           |
|-------------------------|--------------|---------|------------------------------------------|
| `Header`                | 12 B         | 0       | InterfaceId=0x00000001, Mask=STREAM_ID_PROXY, FunctionId=0x00000101 |
| `NumUsbDevice`          | 4 B LE u32   | 12      | **MUST be 0x00000001**                  |
| `UsbDevice`             | 4 B LE u32   | 16      | Client-allocated InterfaceId for this device; used in all subsequent Device-Interface messages |
| `cchDeviceInstanceId`   | 4 B LE u32   | 20      | Count of Unicode (UTF-16LE) chars in `DeviceInstanceId`, INCLUDING the terminating NUL |
| `DeviceInstanceId`      | variable UTF-16LE | 24 | NUL-terminated Unicode string (`cchDeviceInstanceId * 2` bytes) |
| `cchHwIds`              | 4 B LE u32   | ...     | Count of UTF-16LE chars in `HardwareIds` multisz; **MAY be 0** |
| `HardwareIds`           | variable UTF-16LE | ... | multisz (double-NUL terminated) representing the client device hardware IDs. **If `cchHwIds == 0`, buffer MUST NOT be present** |
| `cchCompatIds`          | 4 B LE u32   | ...     | Count of UTF-16LE chars in `CompatibilityIds` multisz |
| `CompatibilityIds`      | variable UTF-16LE | ... | multisz. **If `cchCompatIds == 0`, buffer MUST NOT be present** |
| `cchContainerId`        | 4 B LE u32   | ...     | Count of UTF-16LE chars in `ContainerId` |
| `ContainerId`           | variable UTF-16LE | ... | NUL-terminated Unicode string containing a GUID (MS-DTYP §2.3.4.2). MUST be unique, MUST NOT be zero (all-zero GUID forbidden) |
| `UsbDeviceCapabilities` | **28 bytes fixed** | ... | USB_DEVICE_CAPABILITIES (§2.2.11)   |

- [ ] All `cch*` counters are **CHARACTER counts**, NOT byte counts. Byte length = `cch * 2`.
- [ ] Strings are UTF-16LE with terminating NUL included in the count (standard Windows convention)
- [ ] `cchHwIds` and `cchCompatIds` MAY be zero → the corresponding buffer is absent entirely (not even an empty NUL)
- [ ] `ContainerId`: decode as GUID; reject all-zero GUID (§2.2.4.2 "MUST not be set to zero")
- [ ] Decoder validates `NumUsbDevice == 1`, else reject (malformed → ignore per §3.1.5)
- [ ] On receipt, server creates a device instance keyed by `UsbDevice` (§3.2.5.1.2); duplicate IDs → ignore new msg

### 5.3 `USB_DEVICE_CAPABILITIES` (§2.2.11) — 28 bytes fixed

| Field                                 | Offset | Width  | Type   | Notes                                    |
|---------------------------------------|--------|--------|--------|------------------------------------------|
| `CbSize`                              | 0      | 4 B LE | u32    | **MUST be 28 (0x0000001C)**              |
| `UsbBusInterfaceVersion`              | 4      | 4 B LE | u32    | 0x00000000 / 0x00000001 / 0x00000002     |
| `USBDI_Version`                       | 8      | 4 B LE | u32    | MUST be 0x00000500 or 0x00000600         |
| `Supported_USB_Version`               | 12     | 4 B LE | u32    | 0x0100 (USB1.0) / 0x0110 (USB1.1) / 0x0200 (USB2.0) |
| `HcdCapabilities`                     | 16     | 4 B LE | u32    | **MUST be 0**                            |
| `DeviceIsHighSpeed`                   | 20     | 4 B LE | u32    | 0 = full speed, 1 = high speed. If `UsbBusInterfaceVersion == 0`, this MUST be 0 |
| `NoAckIsochWriteJitterBufferSizeInMs` | 24     | 4 B LE | u32    | 0 = no isoch no-ack support; else jitter buffer ms, **MUST be in [10, 512]** |

- [ ] Total fixed size: **28 bytes**
- [ ] Validator: `CbSize == 28`; `HcdCapabilities == 0`; if nonzero `NoAckIsochWriteJitterBufferSizeInMs` → `10 ≤ v ≤ 512`
- [ ] `NoAckIsochWriteJitterBufferSizeInMs != 0` enables the `NoAck` bit in TS_URB_HEADER for isoch out transfers

## 6. USB Device Interface (§2.2.6) — Server → Client

All messages in this section use: `InterfaceId = ADD_DEVICE.UsbDevice`, `Mask = STREAM_ID_PROXY`.

### 6.1 `CANCEL_REQUEST` (§2.2.6.1)

| Field       | Width | Offset | Value                         |
|-------------|-------|--------|-------------------------------|
| `Header`    | 12 B  | 0      | FunctionId=0x00000100         |
| `RequestId` | 4 B LE u32 | 12 | Identifies the in-flight req |

- [ ] Total size: **16 bytes**
- [ ] RequestId references a prior IO_CONTROL / INTERNAL_IO_CONTROL / TRANSFER_IN_REQUEST / TRANSFER_OUT_REQUEST
- [ ] Client MUST attempt to cancel; if already complete, silently ignore (§3.3.5.3.1)
- [ ] No response PDU is defined — cancellation success is observed via the eventual (or missing) completion

### 6.2 `REGISTER_REQUEST_CALLBACK` (§2.2.6.2)

| Field                  | Width | Offset | Value                                                       |
|------------------------|-------|--------|-------------------------------------------------------------|
| `Header`               | 12 B  | 0      | FunctionId=0x00000101                                       |
| `NumRequestCompletion` | 4 B LE u32 | 12 | 0 → RequestCompletion absent; ≥1 → present                 |
| `RequestCompletion`    | 4 B LE u32 (optional) | 16 | Server-allocated InterfaceId for completion msgs   |

- [ ] Size: **16 bytes** if `NumRequestCompletion == 0`; **20 bytes** if ≥ 1
- [ ] MUST be sent by server once per device **before** any IO_CONTROL / INTERNAL_IO_CONTROL / TRANSFER_*_REQUEST (§3.2.5.3.2)
- [ ] Sending with `NumRequestCompletion == 0` means "stop sending completions" (§3.2.5.3.2, §3.3.5.3.2)
- [ ] Client MUST honor this InterfaceId for all subsequent completions
- [ ] **Implementer enforcement**: if the client receives an IO_CONTROL before any REGISTER_REQUEST_CALLBACK with a nonzero RequestCompletion, treat as protocol violation → close channel

### 6.3 `IO_CONTROL` (§2.2.6.3)

| Field             | Width      | Offset | Notes                                      |
|-------------------|------------|--------|---------------------------------------------|
| `Header`          | 12 B       | 0      | FunctionId=0x00000102                      |
| `IoControlCode`   | 4 B LE u32 | 12     | IOCTL code (§2.2.12)                        |
| `InputBufferSize` | 4 B LE u32 | 16     | Size of `InputBuffer` in bytes             |
| `InputBuffer`     | variable   | 20     | `InputBufferSize` raw bytes                |
| `OutputBufferSize`| 4 B LE u32 | 20+InputBufferSize | Max bytes the client may return  |
| `RequestId`       | 4 B LE u32 | 24+InputBufferSize | Unique ID for this request         |

- [ ] `InputBuffer` MAY be zero length (empty buffer is valid)
- [ ] Valid `IoControlCode` values (§2.2.12): IOCTL_INTERNAL_USB_RESET_PORT, _GET_PORT_STATUS, _GET_HUB_COUNT, _CYCLE_PORT, _GET_HUB_NAME, _GET_BUS_INFO, _GET_CONTROLLER_NAME — implementer need not hardcode codes (passthrough to UrbHandler) but SHOULD log unknowns
- [ ] Client MUST respond with exactly one `IOCONTROL_COMPLETION` matching RequestId (§3.3.5.4.1)

### 6.4 `INTERNAL_IO_CONTROL` (§2.2.6.4)

Same wire layout as IO_CONTROL with `FunctionId = 0x00000103`. Difference: `IoControlCode` is from the Internal IO Control table (§2.2.13), specifically `IOCTL_TSUSBGD_IOCTL_USBDI_QUERY_BUS_TIME = 0x00224000`.

- [ ] Same fields, same offsets as IO_CONTROL
- [ ] Returns IOCONTROL_COMPLETION (not URB_COMPLETION)
- [ ] For QUERY_BUS_TIME: `InputBufferSize = 0`, `OutputBufferSize = 0x4`, response OutputBuffer = u32 LE frame number (1 ms per frame)

### 6.5 `QUERY_DEVICE_TEXT` (§2.2.6.5)

| Field      | Width      | Offset | Notes                                              |
|------------|------------|--------|----------------------------------------------------|
| `Header`   | 12 B       | 0      | FunctionId=0x00000104                              |
| `TextType` | 4 B LE u32 | 12     | Passed to `IRP_MN_QUERY_DEVICE_TEXT` (MSFT-W2KDDK) |
| `LocaleId` | 4 B LE u32 | 16     | LCID                                                |

- [ ] Total size: **20 bytes**
- [ ] **30-second timer** on both sides (§3.1.2): if no response in 30 s, fail with STATUS_TRANSACTION_TIMED_OUT and close the DVC

### 6.6 `QUERY_DEVICE_TEXT_RSP` (§2.2.6.6) — client → server (RESPONSE)

| Field                  | Width         | Offset | Notes                                             |
|------------------------|---------------|--------|---------------------------------------------------|
| `Header`               | **8 B**       | 0      | InterfaceId+MessageId = request's, Mask=STREAM_ID_STUB, **no FunctionId** |
| `cchDeviceDescription` | 4 B LE u32    | 8      | Unicode char count (inc. NUL)                     |
| `DeviceDescription`    | variable UTF-16LE | 12 | NUL-terminated Unicode string                     |
| `HResult`              | 4 B LE u32    | ...    | Operation HRESULT                                 |

- [ ] **Response message** — uses `STREAM_ID_STUB`, no FunctionId in header
- [ ] Header size is 8 bytes (4 for InterfaceId+Mask word, 4 for MessageId)
- [ ] Empty description allowed: `cchDeviceDescription = 1` → single NUL `0x0000`, or `= 0` → absent buffer
- [ ] InterfaceId & MessageId of response MUST equal those of the matching request

### 6.7 `TRANSFER_IN_REQUEST` (§2.2.6.7)

| Field              | Width      | Offset | Notes                                    |
|--------------------|------------|--------|-------------------------------------------|
| `Header`           | 12 B       | 0      | FunctionId=0x00000105                    |
| `CbTsUrb`          | 4 B LE u32 | 12     | Size in bytes of `TsUrb`                 |
| `TsUrb`            | `CbTsUrb` B | 16    | A TS_URB_* structure (§2.2.9)            |
| `OutputBufferSize` | 4 B LE u32 | 16+CbTsUrb | Max input data bytes expected back   |

- [ ] Implementer: validate `CbTsUrb >= 8` (must fit at least TS_URB_HEADER)
- [ ] Validate `CbTsUrb` matches the `TS_URB_HEADER.Size` field on decode
- [ ] Response: **URB_COMPLETION** if any output data, **URB_COMPLETION_NO_DATA** if none (§3.3.5.3.6)
- [ ] `OutputBufferSize` is MAX bytes; client returning more → close DVC (§3.3.5.3.6, §3.2.5.4.2)

### 6.8 `TRANSFER_OUT_REQUEST` (§2.2.6.8)

| Field              | Width      | Offset | Notes                                    |
|--------------------|------------|--------|-------------------------------------------|
| `Header`           | 12 B       | 0      | FunctionId=0x00000106                    |
| `CbTsUrb`          | 4 B LE u32 | 12     | Size of `TsUrb`                          |
| `TsUrb`            | `CbTsUrb` B | 16    | TS_URB_* structure                       |
| `OutputBufferSize` | 4 B LE u32 | 16+CbTsUrb | Size of OutputBuffer in bytes        |
| `OutputBuffer`     | variable   | 20+CbTsUrb | Raw bytes to write to device         |

- [ ] Here `OutputBufferSize` is the EXACT size of the following buffer (not a maximum)
- [ ] Response: always `URB_COMPLETION_NO_DATA`
- [ ] Exception: if `TS_URB_HEADER.A (NoAck)` bit is set AND TS_URB Function is `URB_FUNCTION_ISOCH_TRANSFER` AND USB_DEVICE_CAPABILITIES.NoAckIsochWriteJitterBufferSizeInMs != 0 → **no completion is sent**

### 6.9 `RETRACT_DEVICE` (§2.2.6.9)

| Field    | Width      | Offset | Notes                         |
|----------|------------|--------|-------------------------------|
| `Header` | 12 B       | 0      | FunctionId=0x00000107         |
| `Reason` | 4 B LE u32 | 12     | USB_RETRACT_REASON (§2.2.8)   |

- [ ] Total size: **16 bytes**
- [ ] On receipt, client SHOULD terminate the DVC and stop redirecting the device (§3.3.5.3.8)
- [ ] **`USB_RETRACT_REASON` constants (§2.2.8)**:
  - [ ] `UsbRetractReason_BlockedByPolicy = 0x00000001` — device blocked by server policy
  - [ ] **Only ONE reason is spec-defined**. Implementer: treat unknown reasons as generic retract (still terminate), log the raw value.

## 7. Request Completion Interface (§2.2.7) — Client → Server

All messages: `InterfaceId = REGISTER_REQUEST_CALLBACK.RequestCompletion`, `Mask = STREAM_ID_PROXY` (note: these are NOT response-style — they use STREAM_ID_PROXY and DO carry FunctionId).

### 7.1 `IOCONTROL_COMPLETION` (§2.2.7.1)

| Field              | Width      | Offset | Notes                                              |
|--------------------|------------|--------|----------------------------------------------------|
| `Header`           | 12 B       | 0      | FunctionId=0x00000100                              |
| `RequestId`        | 4 B LE u32 | 12     | Matches `IO_CONTROL.RequestId` or `INTERNAL_IO_CONTROL.RequestId` |
| `HResult`          | 4 B LE u32 | 16     | HRESULT of operation                              |
| `Information`      | 4 B LE u32 | 20     | Bytes transferred / "expected size on overflow"   |
| `OutputBufferSize` | 4 B LE u32 | 24     | Size of `OutputBuffer` (see rules below)          |
| `OutputBuffer`     | variable   | 28     | Result data                                        |

- [ ] `OutputBufferSize` rules (§2.2.7.1):
  - If `SUCCESS(HResult)` → `OutputBufferSize == Information`
  - If `HResult == HRESULT_FROM_WIN32(ERROR_INSUFFICIENT_BUFFER)` (= `0x8007007A`) → `OutputBufferSize == request.OutputBufferSize`, `Information = expected size`
  - Otherwise → `OutputBufferSize MUST be 0`, `Information` is ignored
- [ ] `OutputBufferSize` MUST NOT exceed the request's `OutputBufferSize` → server SHOULD close DVC on violation (§3.2.5.4.1)
- [ ] One-and-only-one per request; duplicates → close DVC
- [ ] Unknown `RequestId` → close DVC

### 7.2 `URB_COMPLETION` (§2.2.7.2)

| Field              | Width      | Offset           | Notes                                       |
|--------------------|------------|------------------|----------------------------------------------|
| `Header`           | 12 B       | 0                | FunctionId=0x00000101                       |
| `RequestId`        | 4 B LE u32 | 12               | Matches `TsUrb.RequestId` of TRANSFER_IN_REQUEST |
| `CbTsUrbResult`    | 4 B LE u32 | 16               | Size of TsUrbResult                         |
| `TsUrbResult`      | variable   | 20               | TS_URB_RESULT_* (§2.2.10)                   |
| `HResult`          | 4 B LE u32 | 20+CbTsUrbResult | HRESULT                                      |
| `OutputBufferSize` | 4 B LE u32 | 24+CbTsUrbResult | Size of OutputBuffer                        |
| `OutputBuffer`     | variable   | 28+CbTsUrbResult | Data read from device                       |

- [ ] Only used when `TRANSFER_IN_REQUEST` actually produced data (otherwise use URB_COMPLETION_NO_DATA)
- [ ] `TsUrbResult` begins with 8-byte `TS_URB_RESULT_HEADER` (Size:u16, Padding:u16, UsbdStatus:u32) — §2.2.10.1.1
- [ ] Implementer: validate `TsUrbResult.Size == CbTsUrbResult` on decode
- [ ] `OutputBufferSize` MUST NOT exceed `TRANSFER_IN_REQUEST.OutputBufferSize` → close DVC (§3.2.5.4.2)

### 7.3 `URB_COMPLETION_NO_DATA` (§2.2.7.3)

| Field              | Width      | Offset           | Notes                                       |
|--------------------|------------|------------------|----------------------------------------------|
| `Header`           | 12 B       | 0                | FunctionId=0x00000102                       |
| `RequestId`        | 4 B LE u32 | 12               | Matches the original request                |
| `CbTsUrbResult`    | 4 B LE u32 | 16               | Size of TsUrbResult                         |
| `TsUrbResult`      | variable   | 20               | TS_URB_RESULT_*                             |
| `HResult`          | 4 B LE u32 | 20+CbTsUrbResult | HRESULT                                      |
| `OutputBufferSize` | 4 B LE u32 | 24+CbTsUrbResult | For TRANSFER_OUT: bytes sent to device. For TRANSFER_IN (no data): MUST be 0. |

- [ ] **If TRANSFER_IN source and `OutputBufferSize != 0` → server closes DVC** (§3.2.5.4.3)
- [ ] If TRANSFER_OUT source and `OutputBufferSize > request.OutputBufferSize` → close DVC

## 8. TS_URB Structures (§2.2.9)

### 8.1 `TS_URB_HEADER` (§2.2.9.1.1) — 8 bytes

| Field          | Width              | Offset | Notes                                        |
|----------------|--------------------|--------|-----------------------------------------------|
| `Size`         | 2 B LE u16         | 0      | Size in bytes of **entire TS_URB** (not just header). MUST equal enclosing `CbTsUrb` |
| `URB Function` | 2 B LE u16         | 2      | URB_FUNCTION_* constant (MSFT-W2KDDK Vol 2 Pt 4 Ch 3) |
| `RequestId`    | **31 bits** (low)  | 4      | Unique request ID                             |
| `NoAck` (A)    | **1 bit** (high)   | 4      | High bit of the 32-bit LE word. See rules     |

- [ ] Fixed 8 bytes
- [ ] RequestId+NoAck packing (LE u32 at offset 4): `word = (RequestId & 0x7FFF_FFFF) | ((NoAck & 1) << 31)`
- [ ] `NoAck` bit:
  - MUST be 0 for TRANSFER_IN_REQUEST
  - MAY be 1 only if: `Function == URB_FUNCTION_ISOCH_TRANSFER` AND `USB_DEVICE_CAPABILITIES.NoAckIsochWriteJitterBufferSizeInMs != 0`
  - When set, client MUST NOT send URB_COMPLETION for that TRANSFER_OUT_REQUEST
- [ ] RequestId is unique across all four IO request types until its completion is sent; then may be reused

### 8.2 Required TS_URB variant types and wire sizes

The implementer needs at least these variants (matching §2.2.9.2–§2.2.9.16). Minimum fixed portions (excluding variable trailers):

| Variant                                      | §         | Fixed header+body size | Notes |

| `TS_URB_CONTROL_TRANSFER`                     | 2.2.9.6   | 24 B                   | PipeHandle:u32, TransferFlags:u32, SetupPacket:[u8;8] |
| `TS_URB_BULK_OR_INTERRUPT_TRANSFER`           | 2.2.9.7   | 16 B                   | PipeHandle:u32, TransferFlags:u32 |

- [ ] Always validate `TS_URB_HEADER.Size` equals the actual structure length, else reject
- [ ] `TS_URB_ISOCH_TRANSFER.NumberOfPackets`: 0 is a valid edge case (empty isoch). Bound-check against a DoS cap (see §12 below)
- [ ] For §2.2.9.15 `Recipient` is the LOW 5 bits of a packed byte; high 3 bits are padding and MUST be ignored on decode; set to 0 on encode

## 12. DoS / Safety Caps — Implementer-Enforced Upper Bounds

MS-RDPEUSB does not prescribe hard caps for most variable-length fields. A malicious server can request enormous transfers/allocations. **Enforce these in `justrdp-rdpeusb` before allocating:**

| Field                                       | Recommended cap    | Rationale                                       |
|----------------------------------------------|--------------------|-------------------------------------------------|
| `TRANSFER_IN_REQUEST.OutputBufferSize`       | **16 MiB**         | Real USB transfers are rarely >1 MB; 16 MiB headroom |
| `TRANSFER_OUT_REQUEST.OutputBufferSize`      | **16 MiB**         | Same                                            |
| `TRANSFER_OUT_REQUEST.OutputBuffer` actual   | exactly OutputBufferSize; reject mismatch | Prevent buffer-length lies         |
| `IO_CONTROL.InputBufferSize`                 | **64 KiB**         | IOCTL inputs are small                          |
| `IO_CONTROL.OutputBufferSize`                | **64 KiB**         | Same                                            |
| `INTERNAL_IO_CONTROL.InputBufferSize`        | **64 KiB**         | Same                                            |
| `INTERNAL_IO_CONTROL.OutputBufferSize`       | **64 KiB**         | Same                                            |
| `CbTsUrb`                                    | **64 KiB**         | Largest real TS_URB (SELECT_CONFIGURATION with many interfaces) is a few KiB |
| `CbTsUrbResult`                              | **64 KiB**         | Same                                            |
| `TS_URB_ISOCH_TRANSFER.NumberOfPackets`      | **1024**           | USB 2.0 limits isoch to ≤1023 packets/URB; 1024 rounds up |
| `TS_URB_ISOCH_TRANSFER_RESULT.NumberOfPackets` | **1024**         | Same                                            |
| `TS_USBD_INTERFACE_INFORMATION.NumberOfPipes` | **64**            | 16 endpoints × IN/OUT is max; 64 generous      |
| `TS_URB_SELECT_CONFIGURATION.NumInterfaces`  | **32**             | USB config descriptor limit                     |
| `ADD_DEVICE.cchDeviceInstanceId`             | **1024 chars**     | Device Instance IDs are typically <200 chars   |
| `ADD_DEVICE.cchHwIds`                        | **4096 chars**     | multisz; Windows convention                     |
| `ADD_DEVICE.cchCompatIds`                    | **4096 chars**     |                                                  |
| `ADD_DEVICE.cchContainerId`                  | **40 chars**       | GUID = 38 chars + NUL + slack                  |
| `QUERY_DEVICE_TEXT_RSP.cchDeviceDescription` | **1024 chars**     | Human-readable description                      |
| Total PDU size (single DVC message)          | **32 MiB**         | Sanity ceiling before processing                 |
| In-flight requests per device                | **1024**           | Prevent server flooding client with fake IDs    |

- [ ] Every decoder that reads a length prefix MUST check against the cap **before** allocating
- [ ] Length prefix × element size MUST NOT overflow usize (use `checked_mul`)
- [ ] On cap violation: log, close the DVC, do NOT panic

## 13. Boundary Cases & Optional Fields

- [ ] `ADD_DEVICE.cchHwIds == 0` → `HardwareIds` buffer absent (skip, don't emit empty buffer)
- [ ] `ADD_DEVICE.cchCompatIds == 0` → `CompatibilityIds` absent
- [ ] `REGISTER_REQUEST_CALLBACK.NumRequestCompletion == 0` → `RequestCompletion` field absent; meaning "stop sending completions"
- [ ] `IO_CONTROL.InputBufferSize == 0` → `InputBuffer` is 0 bytes but still present (buffer of length 0)
- [ ] `TRANSFER_IN_REQUEST` with `OutputBufferSize == 0` → completion uses URB_COMPLETION_NO_DATA
- [ ] `TS_URB_ISOCH_TRANSFER.NumberOfPackets == 0` → IsoPacket array is empty; still valid
- [ ] `URB_COMPLETION.OutputBufferSize == 0` → empty OutputBuffer, still legal (but then NO_DATA should have been used — implementer: accept both for robustness, log oddity)
- [ ] `URB_COMPLETION_NO_DATA` when RequestId was TRANSFER_IN → `OutputBufferSize` MUST be 0; nonzero → close DVC
- [ ] `QUERY_DEVICE_TEXT_RSP.cchDeviceDescription == 0` → no buffer; `HResult` immediately follows `cchDeviceDescription`
- [ ] `CANCEL_REQUEST` may race with completion → if cancel arrives after completion sent, client silently ignores (§3.3.5.3.1)
- [ ] `ADD_DEVICE.NumUsbDevice MUST be 1` — values ≠1 are malformed, ignore per §3.1.5
- [ ] Strings: the `cch*` count INCLUDES the trailing `L'\0'`. A string with a single NUL terminator and no characters has `cch = 1`.
- [ ] Padding bytes in TS_USBD_* structures: encoder MAY emit any value, decoder MUST ignore
- [ ] Highest 3 bits of `Recipient` in §2.2.9.15 decode: MUST be cleared; encode: MUST be 0
- [ ] Top 2 bits of adopted `NetInterfaceId` from QI_RSP: MUST be cleared before use as `InterfaceId`

## 15. State Machine Responsibilities (§3.1, §3.2.5, §3.3.5)

Client side (this is what we're building):

1. [ ] **Channel connected** → wait for `RIM_EXCHANGE_CAPABILITY_REQUEST`
2. [ ] On request → send `RIM_EXCHANGE_CAPABILITY_RESPONSE`
3. [ ] Wait for `CHANNEL_CREATED` (InterfaceId=2 from server)
4. [ ] Validate server version; if unsupported → close DVC
5. [ ] Send own `CHANNEL_CREATED` (InterfaceId=3); enter **Ready**
6. [ ] When host reports a USB device → send `ADD_VIRTUAL_CHANNEL` on control DVC
7. [ ] Server opens a new DVC; on that new channel send `ADD_DEVICE`
8. [ ] Wait for `REGISTER_REQUEST_CALLBACK` (must precede any IO request)
9. [ ] Enter **Device I/O**: process IO_CONTROL, INTERNAL_IO_CONTROL, QUERY_DEVICE_TEXT, TRANSFER_IN_REQUEST, TRANSFER_OUT_REQUEST, CANCEL_REQUEST; emit completions via `RequestCompletion` interface ID
10. [ ] On `RETRACT_DEVICE` → close per-device DVC, stop redirecting
11. [ ] On device unplug → close per-device DVC

- [ ] RequestId allocation is the **server's** responsibility. Client just echoes it back in completions.
- [ ] 30-s timer for QUERY_DEVICE_TEXT (§3.1.2): required on both sides
- [ ] Malformed packet → silently ignore (do NOT terminate) (§3.1.5)
- [ ] Out-of-sequence packet → silently ignore
- [ ] Rule violations flagged as "SHOULD terminate the DVC" in §3.2.5.4/§3.3.5.3/4 → implementer SHOULD terminate

## 16. `UrbHandler` Trait Shape (Mock-friendly)

The PDU crate stays backend-agnostic. Host USB stack integration goes through a trait like:

```rust
trait UrbHandler {
    fn handle_io_control(&mut self, request_id: u32, ioctl: u32, input: &[u8], output_max: u32)
        -> (HResult, u32 /*information*/, Vec<u8> /*output*/);

    fn handle_internal_io_control(&mut self, request_id: u32, ioctl: u32, input: &[u8], output_max: u32)
        -> (HResult, u32, Vec<u8>);

    fn handle_query_device_text(&mut self, text_type: u32, locale: u32)
        -> (HResult, String /*utf-16 source*/);

    fn handle_transfer_in(&mut self, request_id: u32, urb: &TsUrb, output_max: u32)
        -> TransferInResult; // { UrbResult, HResult, Vec<u8> } OR { UrbResult, HResult } (no data)

    fn handle_transfer_out(&mut self, request_id: u32, urb: &TsUrb, data: &[u8])
        -> (TsUrbResult, HResult, u32 /*bytes sent*/);

    fn handle_cancel(&mut self, request_id: u32);

    fn list_devices(&mut self) -> Vec<UsbDeviceDescriptor>; // drives ADD_DEVICE emission
}
```

- [ ] Trait lives in `justrdp-rdpeusb` crate (not PDU crate — needs `alloc`/`std::string`)
- [ ] Provide a `MockUrbHandler` in tests that returns canned results
- [ ] DVC processor owns the trait object and dispatches based on FunctionId
